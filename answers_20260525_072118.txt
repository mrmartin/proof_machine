1. BUFFER SCAN / COUNTER
------------------------
Measured, not estimated. The trainer logs the snapshot time every
round as "synth sampler: freq snapshot in X.Xs (N proofs, T=2.0)":
  - 14.9s at 483K entries (round ~200)
  - 36.2s at 1.15M entries (round 512, this morning)

What it recomputes: ONLY the position-frequency table, from scratch.
synth/freq_counter.py:57-84 opens the JSONL, line-by-line, json-
parses every entry, extracts rule-shape via cert_rule_shape, and
bumps counts[pos][rule]. No dedup, no re-hashing, no cert re-reading
beyond the rule tokens.

Beyond the newest round is rescanned because there is no incremental
persistence. Trivial fix: pickle PositionRuleFreq to disk after each
round, on resume rescan only the JSONL tail past the last persisted
offset. ~30s/round saved -> over 9,500 rounds that's ~80h of wall
time. The README hand-waves this ("a persistent incrementally-
updated counter would erase that cost when needed") — the time is
now, not "around 2-3M entries."


2. WHAT THE SAMPLER ACTUALLY CONDITIONS ON
------------------------------------------
Premise-availability conditioning is genuinely wired. From
synth/backward_gen.py:614-619:

    applicable = [r for r in rules if _applicable(r, steps)]
    if not applicable: break
    rule = _sample_rule(state, len(steps), applicable, rng)

Inverse-frequency softmax runs over the applicable subset only, not
over a static rule list. Both halves of Change 1 shipped.

Caveat: _applicable (lines 514-535) is a necessary-only check (e.g.
">=2 equalities exist" for TRANS, not "two share a term"). The rule
can still fail inside _try_apply's retry loop. So "conditioned on
premise availability" is correct but loose — exactly what item (d)
of the sampler list addresses.


3. NOVELTY METRIC KEYING
------------------------
cert_rule_shape (microgpt_expit.py:186-200) extracts ONLY the ordered
tuple of rule names by scanning for KW_RULE tokens. No witnesses, no
terms.

  - Alpha-renamed witnesses -> identical shape ✓
  - A redundant REFL in the middle -> DIFFERENT shape from the
    no-REFL version ✗

So 28,009 unique shapes is NOT padded by alpha-renaming (the
variant_of fix in 1ca363f was about injective renaming, which is
independent). It IS padded by dead-step variants — ASSUME-REFL-DISCH
and ASSUME-DISCH both prove p ⊢ p but count as two shapes. There is
no shape-canonicalisation pass anywhere. Discount the 28,009 figure
by some unknown factor when reasoning about "actual rule-composition
breadth".

8. KERNEL SANITY PREFIX
-----------------------
The CLAUDE.md note "the OCaml kernel re-verifies the first 200 of
every batch as a sanity prefix" is NOT true for the continuous-
trainer path.

expand_corpus_synthetic (microgpt_expit.py:442-513) has no
KernelClient call, no vpool.verify, no reverify of any kind. The
continuous-trainer synth path appends entries to the buffer based on
Python-shadow correctness alone.

The "first 200" sanity is in synth/build_corpus.py:75 (the offline
corpus builder, separate codepath, kernel-verifies first 50 unless
--reverify is set). That script was not used in the continuous run.

Kernel-verified fraction of the current 1.15M buffer:
  - ~1,600 corpus seeds: verified via curated gold ✓
  - ~64K discovered certs: kernel-verified by vpool.verify_batch ✓
  - ~1.08M synthetic-backward certs: NOT kernel-verified at
    insertion

That is ~95% of the buffer trusting the Python shadow. Commit
1ca363f literally fixed a 0.4% shadow->kernel contamination bug
already. The exposure pattern is the same.

A periodic synth/audit_buffer.py sweep would tell us the current
actual drift rate. If still <=0.4%, fine; if higher, the SFT
distribution has been training on partly-fictional rules for the
last 300+ rounds.
