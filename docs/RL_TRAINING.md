# Training a proof-finding GPT

This document covers the **tokenizer**, the **OCaml kernel verifier
worker**, and the **RL trainer** (`microgpt_with_RL_hol.py`).  The goal is
a small GPT that, given a HOL theorem as prompt, emits a token stream
that decodes to a kernel-verifying certificate of that theorem.

The kernel itself is unchanged — every reward signal is "did the trusted
verifier V accept this?".

## Components

```
HOL term/cert  ──► tokenizer ───► token IDs ───► GPT (microgpt)
                  (OCaml + Python)
                                            │
                                            ▼
                                    grammar-masked sample
                                            │
                                            ▼
                              bin/verify_tokens.exe (subprocess)
                                            │
                                            ▼
                                       reward {-1, 0, +100}
```

### `tokenizer/` — lexeme-level vocab + grammar PDA

- **OCaml library** (`tokenizer/lexicon.ml`, `encode.ml`, `decode.ml`,
  `grammar.ml`, `synth.ml`).  232-token uint8-sized vocabulary.
  Round-trip + grammar-acceptance + mask-soundness tests all green
  (`dune test`).  See [`docs/TOKENIZER.md`](TOKENIZER.md) if it exists,
  or just read the headers — they're short.
- **Python port** (`tokenizer/python/hol_tokenizer.py`).  Mirrors the
  OCaml lexicon byte-for-byte so trainer and verifier agree on tokens.
  Exposes `encode_cert`, `decode_cert`, `encode_term_only`,
  `initial_state`, `step`, `valid_next_mask`, `alpha_eq`, …

### `bin/verify_tokens.ml` — persistent kernel verifier

A line-oriented stdin/stdout worker.  Each line is a verification
request; each response line is the reward integer.

Request format (whitespace-separated):

```
<cert_len> <cert_tok_0> … <cert_tok_{n-1}> <goal_len> <goal_tok_0> … <goal_tok_{m-1}>
```

Response: one integer per line —

- `100` cert kernel-verifies AND `cert.concl ≈α goal`
- `0`   cert kernel-verifies but `cert.concl ≠ goal` (proves the wrong theorem)
- `-1`  cert does not decode, the kernel rejects it, or any other error

The trainer spawns ONE such subprocess per Python rollout-worker (so 48
parallel verifiers under default settings).  Per-request overhead is
~1 ms; no fork+exec on the hot path.

### `microgpt_with_RL_hol.py` — REINFORCE trainer

Karpathy's microgpt **model + Adam left untouched**.  Around it:

- **Data-parallel rollouts**: a `multiprocessing.Pool` with `N_WORKERS`
  workers (default = all CPU cores).  Each RL step samples `N_WORKERS`
  rollouts in parallel on the same prompt, averages gradients, applies
  one Adam update.
- **Grammar-masked sampling**: at every generation step we compute
  `valid_next_mask(state)` and softmax-sample only over legal tokens.
  Every rollout is a syntactically well-formed cert *prefix*.
- **Validator V**: routes to the OCaml verifier subprocess (default).
  A pure-Python fallback (no kernel verify) is kept for the
  end-of-run inference summary.
- **Reward-gated entropy bonus**: workers compute per-step softmax
  entropy `H(π_t)` and add `−β · ΣH` to the loss — but only when the
  rollout failed (`R < 50`).  This keeps solved seeds from drifting
  via entropy pressure while still encouraging exploration where it
  matters.
- **Per-prompt EMA baseline** (`b_label ← 0.9 b_label + 0.1 R̄`).
  REINFORCE with the standard `(R − b)` advantage; gradient computed
  worker-side so main just averages.
- **Optional supervised warmup**: when `HOL_WARMUP_STEPS > 0`, a
  teacher-forced next-token CE pass runs over `HOL_CORPUS_PER_PATTERN ×
  (number of patterns)` examples generated by
  `_build_supervised_corpus`.

### `inspect_ckpt.py`

Loads a `.pkl` checkpoint and prints per-seed greedy + temperature-T
generations.  Useful for post-mortems.

## Run pipeline at a glance

```
seed prompts (7 HOL theorems)
        │
        ▼
prompt-tokenize  → [BOS, encoded-goal, BOS]
        │
        ▼  forward through prompt (KV cache)
        │
        ▼  ┌─ sample 48 rollouts in parallel
        │  │  · mask logits with grammar PDA
        │  │  · stop when grammar accepts (cert closed)
        │  │
        │  ▼
RL loop ─┤  V(goal, cert) via kernel verifier subprocess
        │  │
        │  ▼
        │  per-worker advantage = R − baseline
        │  loss = −adv · Σ log π  − β_eff · Σ H
        │  loss.backward()
        │  │
        │  ▼
        └─ average worker gradients, Adam step
```

## Configuration (env vars)

| Var                      | Default                            | Meaning                                            |
|--------------------------|------------------------------------|----------------------------------------------------|
| `HOL_NUM_STEPS`          | 8000                               | RL steps                                           |
| `HOL_LOG_EVERY`          | 20                                 | RL-step logging period                             |
| `HOL_CKPT_EVERY`         | 200                                | RL-step checkpoint period                          |
| `HOL_NUM_WORKERS`        | `cpu_count()`                      | Parallel rollouts per step                         |
| `HOL_WARMUP_STEPS`       | 0                                  | Supervised warmup steps (0 = skip warmup)          |
| `HOL_CORPUS_PER_PATTERN` | 200                                | Replicas per token-pattern in the warmup corpus    |
| `HOL_ENTROPY_BETA`       | 0.0                                | Entropy bonus weight (set ~0.02 for exploration)   |
| `HOL_USE_KERNEL_VERIFY`  | 1                                  | 1 = kernel verifier subprocess; 0 = pure-Python V  |
| `HOL_VERIFIER_BIN`       | `_build/default/bin/verify_tokens.exe` | Path to the OCaml verifier                     |
| `HOL_BLOCK_SIZE`         | 96                                 | Transformer context length                         |
| `HOL_MAX_GEN`            | 80                                 | Hard cap on generated tokens per rollout           |
| `HOL_REWARD_WRONG_CONCL` | 0                                  | (Python-V fallback) reward for valid-but-wrong concl |
| `HOL_STEP_BONUS`         | 0                                  | (Python-V fallback) bonus if cert has ≥1 step      |
| `HOL_LOG_PATH`           | `hol_rl_run.log`                   | Output log file                                    |
| `HOL_CKPT_PATH`          | `hol_rl_ckpt.pkl`                  | Checkpoint file                                    |

## Quick start

Build the OCaml side (tokenizer library + verifier binary):

```
dune build
```

Run a tiny smoke test (no kernel verify, no warmup):

```
HOL_NUM_STEPS=10 HOL_NUM_WORKERS=4 HOL_USE_KERNEL_VERIFY=0 \
  python3 microgpt_with_RL_hol.py
```

A "production" run mirroring R5 (best so far, 5/7 seeds solved):

```
HOL_NUM_STEPS=4000 HOL_WARMUP_STEPS=200 HOL_CORPUS_PER_PATTERN=200 \
  HOL_ENTROPY_BETA=0.02 HOL_USE_KERNEL_VERIFY=1 \
  HOL_NUM_WORKERS=48 HOL_BLOCK_SIZE=96 HOL_MAX_GEN=80 \
  python3 microgpt_with_RL_hol.py
```

## Run history (summary)

The seeds are seven small HOL theorems exercising the encoded vocabulary:

```
refl_x         (x:nat = x:nat)
refl_y         (y:nat = y:nat)
refl_n         (n:nat = n:nat)
refl_p_bool    (p:bool = p:bool)
refl_q_bool    (q:bool = q:bool)
imp_p          p:bool ==> p:bool
imp_q          q:bool ==> q:bool
```

| Run | Tweaks                                                                                  | Seeds solved | Notes |
|----:|----------------------------------------------------------------------------------------|:------------:|---|
| R0  | Toy alphabet (Karpathy's vowel example).                                                | n/a          | sanity test |
| R1  | HOL vocab + grammar mask; reward `{-1, +1, +100}`.                                      | 0/7          | All converged on 9-token zero-step cheap-+1 cert. |
| R2  | + `V(wrong concl) = 0`.                                                                 | 0/7          | Same plateau at 0 instead of +1. |
| R3  | + `step bonus = +0.5`.                                                                  | 0/7          | All settled at 13-token "has-a-step" but-wrong-concl cert at +0.5. |
| R4  | + 50-step warmup on the 7 seeds.                                                        | 3/7          | refl_x/y/n found correct proofs; mid-run the model migrated to a 17-token *zero-step* cert with the goal copied into the concl (V's syntactic concl-check accepts that). |
| R5  | + OCaml kernel verifier (kills the zero-step cheat) + reward-gated entropy + larger supervised corpus (1600 examples, 6 token-patterns, 200 warmup steps). | **5/7** | refl_x/y/n + refl_p_bool + refl_q_bool all kernel-verify. imp_p / imp_q remain at +0 (the model emits a verifying REFL cert but of the wrong theorem). |

Final R5 inference (greedy + grammar mask) — all 5 successful seeds emit
the exact gold-shape cert (39 tokens) that the kernel accepts:

```
refl_x       gen-len 39  V=+100  ← kernel-verified REFL on v0:nat
refl_y       gen-len 39  V=+100  ← same
refl_n       gen-len 39  V=+100  ← same
refl_p_bool  gen-len 39  V=+100  ← kernel-verified REFL on v0:bool
refl_q_bool  gen-len 39  V=+100  ← same
imp_p        gen-len 39  V=  0   ← kernel-verifies a REFL cert; concl ≠ goal
imp_q        gen-len 39  V=  0   ← same
```

## Knobs to try next

1. **More imp-shape weight in the corpus.**  R5's corpus is 25% imp_self
   but the model still emits REFL for imp prompts.  Going to 50%+ — or
   curriculum-warming-up on imp_self alone first — should push it over.
2. **Bigger model.**  At `n_embd=12, n_layer=1, ~8K params`, the model
   barely has the capacity to differentiate the prompt token streams.
   Bumping `n_embd` to 24–32 or adding a second layer is cheap on CPU
   (scalar autograd is sequential anyway) and should unlock the
   remaining seeds.
3. **Lower `HOL_ENTROPY_BETA`.**  R5 used 0.02; at step 520 a single
   rare grammar-reject leaked enough entropy noise to perturb a solved
   seed (refl_y briefly hit -1).  0.005 with a longer run might be
   more stable.
4. **Increase RL exploration on stuck prompts.**  Right now the per-
   prompt baseline saturates at the current reward, so once imp is
   stuck at +0 the policy gradient is identically zero there.  An
   `epsilon-greedy` schedule (mix in pure-random masked sampling at low
   rate) is a cheap alternative to entropy regularisation.
