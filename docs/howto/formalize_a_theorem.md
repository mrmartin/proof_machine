# How do I... formalize a theorem?

**Persona.** You have an informal mathematical statement on paper —
something like "the sum of two even numbers is even" or "every
non-empty bounded set of reals has a supremum" — and you want to
turn it into a precise object that the kernel can talk about: a
**kernel formula** $\varphi$, stored on disk in a `.kf` file. You
have not yet written its proof; you are just nailing down the
statement.

This persona writes one file: a `.thy` theory file. No OCaml. The
elaborator turns your `.thy` into a `.kf` automatically.

---

## What goes in a `.thy` file

Four kinds of declarations, in order:

| Keyword   | Declares                       | Example                                               |
|-----------|--------------------------------|-------------------------------------------------------|
| `type`    | a type constructor (and arity) | `type nat`                                            |
| `const`   | a typed constant               | `const plus : nat -> nat -> nat`                      |
| `axiom`   | a kernel formula taken on faith | `axiom add_zero : (! n : nat. (plus n one) = n)`     |
| `theorem` | the goal you want to be able to prove later | `theorem refl : (x:nat = x:nat)`        |

Lines starting with `#` are comments. Everything is in ASCII; the
renderer later turns these into proper math symbols when it produces
LaTeX.

---

## Surface syntax

The kernel speaks classical higher-order logic; here is the surface
syntax for terms:

| You write       | It means                                  |
|-----------------|-------------------------------------------|
| `x : T`         | a variable `x` of type `T`                |
| `(f x)`         | application: `f` applied to `x`           |
| `(\ x : T. e)`  | the function $\lambda x. e$               |
| `(a = b)`       | equality                                  |
| `(p /\ q)`      | conjunction                               |
| `(p \/ q)`      | disjunction                               |
| `(p ==> q)`     | implication                               |
| `(~ p)`         | negation                                  |
| `(! x : T. p)`  | $\forall x : T.\ p$                       |
| `(? x : T. p)`  | $\exists x : T.\ p$                       |

Two rules of thumb:

1. **Parenthesise everything**. The parser is intentionally tiny and
   does not infer precedence. `(! n : nat. ((plus n one) = n))` not
   `! n : nat. plus n one = n`.
2. **Annotate the first occurrence of a variable**. After that, the
   parser will pick up its type from the surrounding binder. So you
   can write `(! n : nat. (plus n one))` and the inner `n` does not
   need `:nat`.

## Worked example: formalize "x + 1 ≥ x"

Suppose you want to state the (true) fact: for every natural number
`n`, `n + 1` is at least `n`.

### Step 1 — pick or create a theory package

Look at [`theories/number_theory/theory.thy`](../../theories/number_theory/theory.thy)
to see how an existing package is laid out. You can either extend
that file or make a new one. We will make a new one to keep things
clean.

```
mkdir -p theories/example
```

### Step 2 — write the .thy file

```
cat > theories/example/theory.thy <<'EOF'
# An example theory package — the bare minimum to state x + 1 >= x.

type nat
const one  : nat
const plus : nat -> nat -> nat
const ge   : nat -> nat -> bool

theorem succ_ge : (! n : nat. (ge (plus n one) n))
EOF
```

That's the entire file. Notice:

- `type nat` declares `nat` as a kernel type (arity 0). Inside the
  kernel, `nat` is treated as an opaque base type.
- `const plus : nat -> nat -> nat` says `plus` is a function from two
  naturals to a natural.
- `theorem succ_ge : ...` is the **goal** you want to prove later.
  The right-hand side is the kernel formula.

### Step 3 — elaborate the theorem to a .kf file

```
./_build/default/frontend/elab_main.exe \
    theories/example/theory.thy succ_ge /tmp/succ_ge.kf
```

If you get `elab: succ_ge -> /tmp/succ_ge.kf`, success. Open the
result:

```
;; auto-generated by `elab`; self-contained kernel input
(type nat)
(const one "nat")
(const plus "(nat -> (nat -> nat))")
(const ge "(nat -> (nat -> bool))")
(goal "(! n : nat. ((ge ((plus n:nat) one)) n:nat))")
```

That is the kernel formula. It is now ready for either a prover
backend (see [Prove a theorem](prove_a_theorem.md)) or somebody to
verify a certificate against (see [Verify a proof](verify_a_proof.md)).

## What can go wrong

- **`parse_term: expected . at ...`** — you forgot a `.` after the
  type annotation in a quantifier. Surface form is
  `(! n : nat. body)`, not `(! n : nat body)`.
- **`Axioms.declare: ill-formed axiom X`** — usually a missing
  `const` declaration for a symbol you used in the axiom statement,
  or a type mismatch (e.g. `(ge n m)` where `m : bool` instead of
  `m : nat`).
- **`unknown type constructor: T`** — you referenced a type before
  declaring it. The parser processes `type` declarations before
  `const`/`axiom`/`theorem`, but the type must appear *somewhere* in
  the file.
- **`elab: no theorem named X`** — the second argument to `elab` must
  match the name in `theorem X : ...`. Watch for typos.

## What the elaborator is doing

The frontend (under `frontend/`) is **untrusted**. It:

1. Reads your .thy file line by line.
2. Registers types and constants with the kernel's tables.
3. Parses each axiom and theorem term using
   [`kernel/cert.ml`'s `parse_term`](../../kernel/cert.ml).
4. Writes a self-contained .kf file that, when re-parsed by V, sees
   the same types, constants, axioms, and goal you wrote.

If the elaborator has a bug, the worst that can happen is your .kf
file states a *different* theorem than you meant — and downstream,
the prover and verifier will run on that different theorem. The
kernel cannot be fooled by a buggy elaborator into accepting a
non-theorem.
